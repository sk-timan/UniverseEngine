## 目的

为运行时提供统一的 World/Level/Actor 生命周期抽象，确保关卡加载、切换、逐帧驱动与配置输入链路具备一致性、可追踪性与可验证性。

## 需求

### 需求:系统必须提供 World 级关卡生命周期管理
系统必须提供 `World` 运行时对象，统一负责关卡的加载、释放、切换与逐帧驱动；系统禁止将关卡生命周期控制散落在应用壳层与渲染层的多个入口中。

#### 场景:根据 map_id 加载首个关卡
- **当** 应用完成基础初始化且存在有效 `world.map_id`
- **那么** 系统必须通过 `World` 发起目标关卡加载，并将其设置为当前活动关卡

#### 场景:切换关卡时先卸载旧关卡
- **当** 运行时请求加载与当前不同的关卡
- **那么** 系统必须先完成旧关卡卸载，再完成新关卡加载，禁止两个活动关卡同时处于已激活状态

### 需求:Level 必须管理关卡内 Actor 的创建与释放
系统必须通过 `Level` 管理关卡内 `Actor` 集合及其生命周期状态，必须支持 `Loading/Loaded/Unloading/Unloaded` 的可判定状态流转。

#### 场景:关卡进入 Loaded 后可枚举 Actor
- **当** 关卡状态进入 `Loaded`
- **那么** 系统必须能够读取该关卡内已注册 Actor 集合，并允许逐帧更新流程访问

#### 场景:关卡卸载时清理 Actor
- **当** 关卡进入 `Unloading` 并完成卸载
- **那么** 系统必须释放该关卡 Actor 的活动引用，并使其不再参与后续 Tick

### 需求:Actor 必须提供统一基础语义
系统必须提供 `Actor` 基类，统一定义标识、名称、变换、更新与销毁语义；系统禁止在同一关卡内使用无统一生命周期协议的临时场景元素对象替代 Actor。

#### 场景:Actor 参与逐帧更新
- **当** `World` 进入正常帧循环且活动关卡为 `Loaded`
- **那么** 系统必须按关卡 Actor 集合驱动更新流程，并保持未激活或待销毁 Actor 不执行更新

#### 场景:Actor 标记销毁后退出生命周期
- **当** Actor 被标记为待销毁并完成回收流程
- **那么** 系统必须使该 Actor 不可再被关卡检索为活动对象

### 需求:map_id 必须形成配置到关卡实例的闭环
系统必须将 `gameplay` 配置中的 `world.map_id` 作为关卡选择输入，并确保该输入在关卡加载链路中可追踪；系统禁止忽略 `map_id` 而直接硬编码固定关卡。

#### 场景:map_id 缺失或无效时阻止关卡激活
- **当** `map_id` 缺失、为空或映射失败
- **那么** 系统必须拒绝激活目标关卡，并返回可定位问题的错误信息

#### 场景:map_id 有效时完成实例绑定
- **当** `map_id` 有效且映射存在
- **那么** 系统必须创建并绑定对应关卡实例，供后续渲染与玩法系统使用
