## 上下文

当前 `OpenSpecTest` 已具备 Qt 编辑面板、`ResourceLoader` 与运行时加载链路，但编辑面板仍是占位 UI，无法真实修改并落盘 `gameplay` 配置。  
同时，`gameplay.json` 的现有结构（嵌套 `world/economy`）与加载逻辑（历史上偏向手写字段解析）存在潜在不一致，容易造成“能写不能读”或“读写语义不一致”问题。

本次设计围绕一个目标：建立单一可信配置路径，使 UI、内存模型与文件格式保持一致，并支持后续能力（例如 fish 配置编辑、热重载）平滑扩展。

## 目标 / 非目标

**目标：**
- 将 `gameplay` 配置变为可加载、可编辑、可校验、可保存的闭环流程。
- 使用通用 JSON 库（当前工程已接入 `nlohmann/json`）统一读写实现，避免继续扩散手写字符串解析。
- 固定并明确 `gameplay.json` schema：`version`、`world.map_id`、`world.spawn`、`economy.starting_coins`。
- 提供 UI 侧“应用 / 重置 / 脏状态”交互，保证用户操作可预期。
- 对旧格式读取提供兼容回退，保存时统一收敛到新 schema。

**非目标：**
- 不实现 fish CSV 的可视化编辑器。
- 不实现运行时自动热重载（保存后即时生效可后续扩展）。
- 不引入复杂撤销/历史版本系统。
- 不改变渲染/输入逻辑，仅涉及配置与编辑器流程。

## 决策

### 决策 1：采用通用 JSON 库作为配置读写唯一实现
- 选择：在配置读写中使用 `nlohmann/json`，替代手写解析和 Qt JSON 路径。
- 原因：工程已完成库接入并在构建系统中可用；可将数据层与 UI 框架解耦；读写语义一致且便于测试。
- 备选方案：
  - 继续手写解析：改动小，但长期维护风险高、schema 漂移概率高。
  - Qt JSON：实现可行，但会让数据层额外依赖 Qt 类型体系，不利于后续在非 Qt 场景复用。

### 决策 2：抽出配置存储/校验层，UI 不直接操作文件
- 选择：新增独立存储层（如 `GameplayConfigStore`），对外提供 `Load/Validate/Save`。
- 原因：解耦 UI 与文件格式细节，便于后续测试与复用。
- 备选方案：
  - 在 `MainWindow` 中直接读写文件：实现快，但状态管理与错误处理会迅速复杂化。

### 决策 3：保存统一写新 schema，读取支持旧格式回退
- 选择：读取时优先新格式，必要时回退旧字段；保存仅写新格式。
- 原因：兼顾兼容性与长期一致性，避免多 schema 并存。
- 备选方案：
  - 强制仅支持新格式：迁移成本高，易导致旧数据直接失效。
  - 长期双写双读：逻辑复杂且容易遗漏。

### 决策 4：先实现 gameplay 主配置，不同时引入 fish 编辑
- 选择：以 `gameplay` 字段为首批范围，fish 留到下一变更。
- 原因：控制变更体量，降低交叉风险，加快第一版可用闭环交付。
- 备选方案：
  - 一次覆盖 gameplay + fish：功能完整但复杂度高，验证面扩大。

## 风险 / 权衡

- [schema 差异导致历史数据加载异常] → 增加读取回退逻辑；对缺失字段给出默认值与可读错误。
- [UI 状态与内存模型不同步] → 引入“loaded model / draft model”双态；以显式 `Apply/Reset` 驱动状态转换。
- [校验规则不一致导致运行期失败] → 将校验集中在存储层；UI 只展示错误，不重复定义业务规则。
- [通用 JSON 与现有数据结构映射出错] → 增加字段级单元测试与样例文件回归测试（新格式 + 旧格式）。
- [后续扩展时能力边界不清] → 在本次设计中明确非目标，后续按独立 capability 拆分。

## 迁移计划

1. 新增 `gameplay` 配置存储层与统一校验逻辑。  
2. 重构 `ResourceLoader` 中 `gameplay` 加载路径，接入统一读写/解析。  
3. 将构建与包含路径统一到已接入的 `nlohmann/json`（移除数据层对 Qt JSON 的依赖假设）。  
4. 改造 `MainWindow` 编辑面板控件绑定与交互（加载、应用、重置、脏状态）。  
5. 添加旧格式读取兼容并通过保存收敛新 schema。  
6. 手动验证：修改并保存后重启一致、非法输入被拦截、旧格式可被迁移。  

回滚策略：若出现高风险回归，可临时退回“只读加载”模式并保留旧路径开关（不删除旧数据文件）。
