## 新增需求

### 需求:编辑面板必须提供 gameplay 配置读写闭环
系统必须在 Qt 编辑面板中提供 `gameplay` 配置的加载、编辑、校验与保存能力，并确保保存后文件内容与界面状态一致。

#### 场景:启动后加载并显示当前配置
- **当** 应用启动且 `gameplay.json` 文件有效
- **那么** 系统必须将 `version`、`world.map_id`、`world.spawn`、`economy.starting_coins` 显示到对应编辑控件

#### 场景:用户应用修改后保存成功
- **当** 用户修改上述字段并点击“应用”，且所有字段通过校验
- **那么** 系统必须将新配置写入 `gameplay.json`，并将界面状态标记为“无未保存更改”

### 需求:配置读写必须统一使用通用 JSON 库
系统必须使用同一套通用 JSON 读写实现（`nlohmann/json`）处理 `gameplay` 配置，禁止在同一能力内混用手写字符串解析或 Qt JSON 路径。

#### 场景:保存时输出统一 schema
- **当** 用户执行保存操作
- **那么** 系统必须按统一嵌套 schema 写出文件（包含 `world` 与 `economy` 节点），并禁止写出不一致的扁平结构

#### 场景:读取时兼容旧格式
- **当** 文件缺少新 schema 字段但包含旧字段形式
- **那么** 系统必须按兼容策略完成读取，并在下一次成功保存时收敛为统一 schema

### 需求:保存前必须执行字段级校验并反馈错误
系统必须在保存前对关键字段执行校验，并禁止将非法数据写入文件。

#### 场景:校验失败时阻止保存
- **当** `map_id` 为空，或 `starting_coins` 小于 0，或 `spawn` 任一值非有效数值
- **那么** 系统必须拒绝保存，并向用户反馈可定位到字段的错误信息

#### 场景:校验通过时允许保存
- **当** 所有必填字段存在且数值合法
- **那么** 系统必须允许保存并展示成功反馈

### 需求:编辑器必须支持重置与脏状态提示
系统必须在配置编辑过程中跟踪未保存更改，并提供重置能力恢复到最近一次成功加载或保存的状态。

#### 场景:存在未保存更改时显示脏状态
- **当** 用户修改任一配置字段且尚未保存
- **那么** 系统必须显示“有未保存更改”状态，并启用“应用/重置”操作

#### 场景:执行重置后恢复一致状态
- **当** 用户点击“重置”
- **那么** 系统必须将控件值恢复到最近已保存配置，并清除脏状态提示
