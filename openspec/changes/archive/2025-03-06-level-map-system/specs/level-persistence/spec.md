## 目的

为 Level 提供序列化与反序列化能力，支持关卡数据的持久化保存与恢复，确保 World/Level/Actor 生命周期系统的完整性。

## 需求

### 需求：Level 必须提供序列化接口

Level 必须提供 Save/Load 接口，将关卡内 Actor 数据转换为可持久化格式，并在加载时恢复 Actor 状态。

#### 场景：保存当前关卡数据

- **当** 用户请求保存当前活动关卡
- **那么** 系统必须将当前关卡内所有 Actor 的标识、类型、Transform、组件数据序列化为目标文件格式

#### 场景：加载关卡数据并恢复 Actor

- **当** 用户请求加载关卡文件
- **那么** 系统必须反序列化文件数据，并在当前 Level 中恢复对应 Actor 的状态

### 需求：序列化格式必须可读可扩展

系统必须使用 JSON 作为 MVP 阶段的序列化格式，确保数据可读且易于调试。

#### 场景：JSON 格式结构

- **当** 系统序列化 Level 数据
- **那么** 输出 JSON 必须包含：level_id、actors 数组、metadata 对象

#### 场景：加载损坏或格式错误的文件

- **当** 加载的 JSON 文件格式错误或数据不完整
- **那么** 系统必须返回清晰的错误信息，拒绝加载并保持当前状态不变

### 需求：Actor 序列化必须保留关键状态

系统必须为 Actor 序列化关键运行时状态，确保加载后可以恢复基本场景结构。

#### 场景：Actor 基础数据序列化

- **当** 序列化 Actor
- **那么** 必须保留：actor_id、name、type、transform (position, rotation, scale)

### 需求：Level 卸载与重新加载必须保持数据一致性

系统必须在 Level 卸载后能够正确重新加载已保存的数据。

#### 场景：卸载后加载相同关卡

- **当** 先保存关卡，然后卸载，再加载相同关卡文件
- **那么** 加载后的 Actor 集合应与保存时一致

### 需求：World 必须管理关卡定义与持久化数据的关联

系统必须支持在 World 级别管理关卡定义与对应持久化数据的关联。

#### 场景：注册带持久化数据的关卡

- **当** 注册新关卡定义时
- **那么** 系统必须能够关联对应的持久化数据，并在需要时进行保存/加载
