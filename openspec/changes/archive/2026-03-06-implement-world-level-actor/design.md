## 上下文

当前工程已具备 Win32 + DX12 渲染主循环、Qt 驱动帧更新与基础数据加载能力，但场景组织仍以 `GameApp` 直接驱动渲染器为主，缺少关卡生命周期抽象。`gameplay` 配置中已有 `world.map_id`，但尚未形成“配置 -> 关卡实例 -> 场景元素”的统一运行时模型。

本次变更的约束如下：
- 需与现有 `GameApp`、`Dx12Renderer`、`ResourceLoader`、`ObjectRegistry/UObject` 协同，不破坏现有 MVP 可运行路径。
- 首版优先单机离线与可解释架构，不引入额外外部依赖。
- 为后续钓鱼状态机、鱼类内容、可钓区域与交互对象预留扩展点。

主要相关方：
- 游戏玩法迭代：需要可扩展的实体载体（鱼点、热点、触发器）。
- 渲染/性能迭代：需要可控的加载与释放边界。
- 配置与工具链：需要 `map_id` 可稳定驱动关卡装载。

## 目标 / 非目标

**目标：**
- 建立 `World -> Level -> Actor` 三层运行时结构，统一加载、Tick 与释放语义。
- 将 `map_id` 纳入关卡加载入口，形成稳定生命周期闭环。
- 限定模块职责边界，避免渲染器承担世界管理职责。
- 与现有对象系统衔接，支持对象可追踪与安全回收。

**非目标：**
- 不在本变更中完成异步流式加载、多关卡并行驻留或跨线程调度。
- 不在本变更中实现完整鱼类行为、钓点逻辑或复杂物理系统。
- 不在本变更中重构所有渲染数据结构，仅完成必要接入。

## 决策

### 决策 1：采用 `World -> Level -> Actor` 固定层级
- 选择：`World` 管全局生命周期与活动关卡，`Level` 管关卡内实体集合，`Actor` 管实体行为与状态。
- 原因：与游戏运行时模型直观对应，便于后续按关卡拆分数据与逻辑。
- 备选方案：
  - 仅引入 `Level + Actor`：会把全局生命周期逻辑继续堆在 `GameApp`，长期耦合高。
  - ECS 一次性落地：当前 MVP 复杂度过高，迁移成本与认知成本都偏大。

### 决策 2：首版仅支持单 Active Level，同步加载
- 选择：`World` 在任一时刻只持有一个活动关卡；加载/卸载采用同步路径。
- 原因：最小可行复杂度，先验证架构正确性与生命周期稳定性。
- 备选方案：
  - 多关卡并行：需引入更复杂的资源与可见性管理，超出本阶段收益。
  - 异步加载：需线程安全、状态协调与回滚方案，适合后续性能阶段。

### 决策 3：`Actor` 复用现有 `UObject/ObjectRegistry` 能力
- 选择：`AActor` 作为对象系统中的可管理对象，使用统一 ID/引用关系。
- 原因：避免重复发明生命周期与回收机制，降低内存泄漏风险。
- 备选方案：
  - 独立 new/delete 管理：实现快，但缺少统一追踪与图关系管理。

### 决策 4：`Level` 描述资源需求，渲染资源创建仍归渲染层
- 选择：关卡只维护“需要什么”（模型路径、渲染代理描述），不直接持有 DX12 设备细节。
- 原因：隔离玩法域与渲染域，减少未来替换渲染实现的影响面。
- 备选方案：
  - 让 Actor 直接操控渲染器：短期方便，长期耦合与测试成本高。

### 决策 5：`GameApp` 作为驱动壳，世界成为主业务入口
- 选择：`GameApp.Tick()` 改为驱动 `World.Tick()`，并通过世界查询向渲染层提交需要绘制的数据。
- 原因：让应用层维持“编排者”角色，避免业务逻辑继续扩散到 App 壳层。
- 备选方案：
  - 保持 `GameApp` 直连渲染与逻辑：会阻碍后续系统拆分。

## 风险 / 权衡

- [关卡切换瞬间资源释放不完整] -> 在 `Level` 引入明确状态机（Loading/Loaded/Unloading/Unloaded）与两阶段销毁，先逻辑解绑再资源回收。
- [对象系统接入不当导致引用悬挂] -> 约束 Actor 间引用通过对象 ID/注册表访问，切换关卡时移除 Level 根引用并做集中回收验证。
- [首版同步加载带来卡顿] -> 作为已知权衡接受；在接口层预留异步入口（不在本变更实现）。
- [渲染与世界边界不清] -> 通过“Level 只声明资源需求、Renderer 只做渲染执行”的职责文档与代码接口约束降低漂移。

## Migration Plan

1. 新增 `world` 模块骨架（`UWorld`、`ULevel`、`AActor`）与最小状态机。
2. 在 `GameApp` 初始化阶段创建 `UWorld`，由 `gameplay.map_id` 触发首个关卡加载。
3. 将帧循环改为 `GameApp -> World -> Level -> Actor`，保留现有渲染循环。
4. 让 `Dx12Renderer` 消费关卡提供的场景输入（先覆盖当前地图模型路径场景）。
5. 完成关卡卸载路径与退出路径验证，确保对象与资源释放。
6. 验证通过后，逐步接入后续实体类型（可钓区域、鱼点、调试 Actor）。

回滚策略：
- 保留旧的 `GameApp` 直连渲染入口作为临时兜底分支；若出现严重稳定性问题，可回退到旧路径并禁用世界模块入口。

- 首版 `Actor` 最小公共组件是否只包含 Transform，还是同时纳入轻量标签/分类能力？
- 渲染提交边界采用“World 直接推送”还是“Renderer 主动拉取可见代理”哪种更适合后续扩展？
- 关卡切换时玩家状态（位置、相机、背包上下文）的保留策略是否需要在本变更内定义最小协议？
